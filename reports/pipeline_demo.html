Pipeline shape

TradeAlgo snapshot ─┐ (cached JSON, 3 MB) │ ├──► parse_movement_darkflow ─► darkflow_zscores ─► top_N │ │ │ ▼ Polygon daily bars ──┼──► compute_momentum_signal ──────────► momentum_z ─┤ Polygon options chain ─┼──► compute_flow_signal ──────────► flow_z ─┤ Polygon stock trades ─┘ │ Polygon options IV ───► compute_sentiment_signal ──────────► sentiment_z ─┤ │ TickerSignal records ◄────────────┘ │ ▼ CategoryAnalyzer.analyze() (Claude tool_use) │ ▼ {categories: [{name, tickers, direction, confidence, narrative, ...}]}

What the pipeline does end-to-end

  1. Acquire — once per trading day (4:35 PM ET cron, not yet wired), TradeAlgoClient.fetch_snapshot() GETs the 3 MB JSON bundle and caches it atomically per UTC date. Provides dark-flow movers across small/mid/large caps plus a rolling history.
  2. Distill dark flow — parse_movement_darkflow flattens the three movement files (dedupes by max multiplier) into DarkFlowRecords. darkflow_zscores computes a per-ticker cross-sectional intensity z combining z(multiplier), z(log dollar_value), and z(ats_dollar_volume_pct).
  3. Compose signals — for the top-N (by dark-flow dollar volume), the compass signal layer fetches real Polygon data and computes four z-scores: momentum_z (returns / breakout / vol-adj momentum), flow_z (OI / P/C / vol-OI / large prints; optionally folds in darkflow_z), sentiment_z (25d skew / term structure / IV rank), and darkflow_z itself. Every value is real or None — no fallbacks.
  4. Categorize — CategoryAnalyzer.analyze() builds a CSV prompt of the per-ticker z-scores and asks Claude (claude-opus-4-7) via tool_use with a strict JSON schema to label 3–5 coherent market themes with name, ticker list, direction, confidence, and narrative. The LLM never sees prices and never picks trades — it labels signals the engine already computed. Output is cached idempotently per (asof_date, prompt_hash).
  5. Consume — downstream readers (daily memo, risk dashboards, the strategy layer) ingest data/llm_analysis/{date}.json and either render the categories or filter trade candidates by theme.

What the live LLM call validates

  • Claude correctly identified the most diagnostic pattern on the tape: momentum + dark flow positive, sentiment deeply negative ("smart-money buying into bearish chatter") — a cross-signal divergence a single-feature engine would miss.
  • darkflow_z earned its keep — it is specifically cited in 4 of 5 narratives (META flow + dark, ADTN +0.786 hidden accumulation, ILMN/LUMN negative = "no institutional defense", IREN/ASTS dark positive).
  • The confidence-flag contract worked: categories at 0.45 and 0.42 self-rated low-conviction and explicitly flagged it in their narratives per the prompt's hard rule. No category was padded to hit 5.
  • Categories cross GICS boundaries by design — Category #1 spans social / mobile ad-tech / satellite broadband / crypto&AI hosting under one capex thesis, exactly the kind of CROSS-sector grouping the system prompt asks for.

Caveats logged honestly

  • GICS sector / industry is not yet wired into the compass pipeline — Claude was given empty sector fields and inferred themes from ticker symbols + the multi-signal pattern. Once GICS is plumbed in, expect tighter cluster fit (and higher confidence) on sector-aligned themes.
  • Composite reweighting in compass.signals.flow_proxy is still opt-in (darkflow_z=None by default), so today's live trading paths (EXP-3303b / 3309 / 3311) are unaffected. Promoting dark-flow into the live composite requires a backtested weight vector first.
  • The CategoryAnalyzer output is idempotent per (asof_date, prompt_hash) — re-running this demo without changing inputs hits the cache, not the LLM.
