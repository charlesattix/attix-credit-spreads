Attix Credit Spreads A Complete Architectural Overview — Written for Everyone

Last updated: 2026-05-24 9 active experiments 12,607 tests ~440 modules

1 The Big Picture — What Is This?

Attix Credit Spreads is an automated stock-options trading robot that runs 24/7 on a cloud server. Its job is simple: find good moments to sell options contracts and collect cash (called a "credit"), then manage those positions until they expire profitably or need to be closed early.

What is a Credit Spread? (2-minute explanation)

Imagine you're a bookmaker. Someone wants to bet that SPY (the S&P 500 index fund) will crash below $500 in the next 30 days. You think that's very unlikely, so you take the other side of that bet and collect money upfront.

In options terms:

  • You sell a put option at $505 (the "short strike") — you collect cash for this
  • You buy a put option at $495 (the "long strike") — you pay a little cash for this insurance
  • The difference is your credit — the money you keep if SPY stays above $505
  • Your maximum loss is capped at $10 per share (the spread width) minus the credit you collected

This structure is called a bull put spread. There's also a bear call spread (you bet the stock won't rise too much) and an iron condor (you combine both sides — betting the stock stays inside a range). The system trades all three, picking the right one based on what the market is doing.

Why options selling works statistically
The market consistently overprices the cost of insurance (implied volatility is usually higher than realized volatility). Options sellers collect this "volatility risk premium." Over thousands of trades, this edge compounds. The system targets a backtest Sharpe ratio of 6.39 (elite-level), with an expected live Sharpe of 3.2–4.5 after accounting for real-world slippage and execution costs.

The System's Goal

Run 9 simultaneous experiments (strategy variations) in parallel paper-trading accounts on Alpaca's brokerage platform, track every trade, manage risk automatically, and generate daily performance reports — all without any human clicking buttons.

2 How It Starts — From Server On to First Scan

The system runs on Railway, a cloud hosting platform (think of it like renting a computer that runs your code forever). When Railway starts the application, here's the exact chain of events:

1
Railway reads the Procfile
The file Procfile at the project root tells Railway what processes to start: 
web: uvicorn web_dashboard.app:app --host 0.0.0.0 --port $PORT
worker: python railway_worker.py
watchdog: python railway_watchdog.py
Three processes launch simultaneously: a web dashboard, the main trading worker, and a health watchdog.
2
railway_worker.py boots up
File: railway_worker.py. This is the master conductor. It reads the experiment registry (experiments/registry.json) to find all experiments with "status": "active". Currently there are 9 active experiments. 
from experiments.manager import get_manager
active = get_manager().active()
# Returns 9 experiments: EXP-400, EXP-401, EXP-503,
# EXP-600, EXP-800, EXP-1220, EXP-3303b, EXP-3309, EXP-3311
3
One subprocess per experiment
For each active experiment, the worker spawns a separate Python process, staggered 2 seconds apart to avoid log collisions: 
for p in procs.values():
    p.start()
    time.sleep(2)  # stagger so logs don't collide
Each subprocess runs: python main.py scheduler --config configs/paper_expNNN.yaml --db data/pilotai_expNNN.db

Each experiment gets its own Alpaca API credentials (injected as environment variables like ALPACA_API_KEY_EXP400) and its own separate database file — so they're completely isolated.
4
main.py initializes the trading system
Inside each subprocess, main.py calls create_system(config_file) which:
  • Loads and validates the YAML config for this experiment
  • Checks paper-mode safety (refuses to accidentally trade with real money)
  • Connects to Alpaca's paper-trading API
  • Runs a startup reconciliation (checks if any trades from last session are still open)
  • Pre-warms the data cache with SPY, ^VIX, TLT price history
5
The ScanScheduler starts and the PositionMonitor launches as a background thread
Two things start simultaneously:
  • ScanScheduler (shared/scheduler.py): Like an alarm clock with 18 alarms per trading day. It sleeps between scheduled times and wakes up to run scans.
  • PositionMonitor (execution/position_monitor.py): A background daemon that checks every 60 seconds whether any open positions need to be closed (stop loss hit, profit target reached, etc.)
6
First scan fires
The scheduler waits until the next scheduled slot, then calls system.scan_opportunities(). The first slot each day is at 9:00 AM ET (a pre-market status check), followed by the first real scan at 9:15 AM ET.

3 The Scan — What Happens Every 30 Minutes

When the scheduler fires a scan slot, it calls system.scan_opportunities() in main.py. Think of this as the system's brain doing a complete market check.

Step 1: Entry gates — should we even scan right now?

Before touching any market data, the scan checks a series of gates that might block it entirely:

  • NFP filter (EXP-3311): If today is the day before a Non-Farm Payrolls jobs report (the biggest monthly economic announcement), skip all new entries. The jobs report creates violent market moves.
  • Execution window gate (EXP-3309): If configured, only allow trades during the 15:30–16:00 ET window (the final 30 minutes before close), when options liquidity is highest. All scans outside this window return zero opportunities.

Step 2: Build the scan universe — which tickers to scan?

The system determines which stocks/ETFs to scan. Most experiments scan a fixed list from their config (usually just SPY). The COMPASS system optionally ranks sector ETFs (XLF, XLI, GLD, SLV, etc.) by their momentum quadrant and adds them dynamically.

Step 3: Parallel ticker analysis

The system analyzes up to 4 tickers simultaneously using a thread pool. For each ticker, it runs _analyze_ticker(): 

with ThreadPoolExecutor(max_workers=4) as executor:
    futures = {
        executor.submit(self._analyze_ticker, ticker, direction_override, rrg_quadrant): ticker
        for ticker, direction_override, rrg_quadrant in ticker_universe
    }

Step 4: Per-ticker analysis pipeline

For each ticker, the following chain runs in order:

  1. Fetch 2 years of daily price data — needed for moving average calculations (MA200 requires 200 days of warmup)
  2. Fetch the live options chain — all available strike prices and expirations
  3. Technical analysis — calculates RSI, moving averages, support/resistance levels
  4. Regime detection — determines if the market is bull, bear, neutral, or transitioning (see Section 4)
  5. Regime gate check — for SPY/QQQ, blocks entries during "transition" or "high_stress" regimes
  6. IV rank calculation — measures current volatility vs. the last year (high IV rank = expensive options = good time to sell)
  7. Signal generation — strategy classes generate specific trade proposals (see Section 4)
  8. Scoring — each signal scored 0–100 using 5-component system
  9. Re-pricing — signals priced against real options chain data (no synthetic prices ever)
  10. Return opportunities — the list of viable trades for this ticker

Step 5: Sort, alert, and route

All opportunities from all tickers are merged and sorted by score (highest first). The top 5 are logged. Any opportunity scoring ≥ 60 gets sent as a Telegram notification. All opportunities are then pushed through the AlertRouter, which applies the final risk gate and submits approved trades to Alpaca.

4 Signal Generation — How the System Reads the Market

Regime Detection: Is the market bullish, bearish, or uncertain?

The most critical decision the system makes before any trade is: what mood is the market in right now? This is called regime detection. The system uses the ComboRegimeDetector, which combines three independent signals:

SignalWhat it measuresBull condition
Moving Average (MA80) Is SPY's price above or below its 80-day average price? Price > MA80 (trending up)
RSI (Relative Strength Index) Is buying momentum strong or is the market oversold? RSI > 50 (buyers winning)
VIX Structure Is the market more scared of near-term or far-term risk? (VIX vs VIX3M ratio) VIX < VIX3M (normal, calm structure)

The three signals vote and the regime label is assigned:

RegimeMeaningTrading action
bullAll signs point up — trend is healthy, fear is lowSell bull put spreads (bet market stays up)
bearTrend is down, fear is elevatedSell bear call spreads (bet market stays down)
neutralMixed signals — no clear directionSell iron condors (bet market stays flat)
transitionRegime is changing, high uncertaintySkip all entries (EXP-3303b gate blocks SPY/QQQ)
high_stressExtreme market fear or dislocationSkip all entries for SPX-sensitive tickers
Critical safety rule
If regime detection fails (returns None), the system does not default to any regime. Instead, it logs an error and skips all entries for that ticker entirely. Missing regime data must block trading, not allow it.

if regime is None and combo_mode_active: return [] # Hard block

VIX Check: How scared is the market?

VIX is the "fear index" — it measures how much the market is paying for crash insurance. The system checks VIX at multiple points:

  • VIX > 40: Hard block — no new entries at all (EXP-800). Market is panicking.
  • VIX term structure: When short-term VIX exceeds long-term VIX3M (inverted), that's a warning sign used in regime detection above.

The 5-Component Signal Score (0–100)

Once a candidate trade is generated, it gets scored by shared/signal_scorer.py using five criteria:

ComponentMax PointsWhat it rewards
Credit quality25High credit collected relative to spread width
Risk/reward ratio25Credit received vs. maximum possible loss
Probability of profit25How far OTM (out-of-the-money) the short strike is
Technical alignment15Does the trade direction match the trend?
IV rank10Is implied volatility historically high? (Good time to sell)

Only signals scoring ≥ 60 get Telegram alerts. The AlertRouter applies additional risk gates before placing any actual order.

5 Trade Selection — Picking the Exact Contracts

Once the system decides "yes, sell a bull put spread on SPY," it needs to pick the exact option contracts. This requires choosing three things: expiration date, strike price, and spread width.

DTE — Days to Expiration

Each experiment is configured with a target DTE (how many days until the options expire). For example, EXP-800 uses DTE=15. The system scans available expirations and picks the one closest to the target. Shorter DTE = faster theta decay (time value erodes faster) but less time to recover if the trade goes against you.

ExperimentDTERationale
EXP-400 (Champion)15Sweet spot for SPY — fast decay, manageable risk
EXP-401 (Blend)15–28Straddles need slightly more time
EXP-600 (IBIT Bitcoin)14Bitcoin moves fast — shorter DTE to limit exposure
EXP-1220 (Leverage 3x)Configured in YAMLFollows EXP-400 signal generation

Strike Selection — How Far OTM?

"OTM%" (out-of-the-money percentage) controls how far the short strike is from the current price. A 2% OTM means if SPY is at $500, the short strike would be around $490 (2% below current price).

The system uses two methods to find the right strike:

  • Fixed OTM%: Simple — calculate the target price from a percentage offset. EXP-400 uses 2% OTM for its put spreads.
  • Delta targeting: More sophisticated — finds the strike where the option has a specific delta (probability of expiring in-the-money). This is done in shared/strike_selector.py using Black-Scholes mathematics (pure Python, no external dependencies): 
    # Find the strike closest to delta=0.20 (20% chance of loss)
target_delta = 0.20
best_strike = min(available_strikes,
    key=lambda k: abs(bs_delta(k, S, T, r, sigma) - target_delta))

Spread Width

The spread width is the dollar distance between the two strikes (e.g., $5, $10, $12). Wider = more credit collected but also more max loss. EXP-400's champion config uses $12-wide spreads. The width is configured per experiment and fixed — the system doesn't dynamically adjust it.

6 Position Sizing — How Much Money per Trade?

Even a great trade can be ruined by betting too much on it. The system uses several layers of sizing logic to determine how many contracts to trade.

Kelly Criterion — The Mathematical Bet-Sizing Formula

The Kelly Criterion is a mathematical formula originally developed for gambling that answers: "Given your edge and win rate, what fraction of your bankroll should you bet?" Betting too much → ruin. Betting too little → leaving money on the table.

EXP-800 ("Safe Kelly 4/7/9") implements regime-adjusted Kelly fractions:

RegimeKelly FractionMeaning
Bull9%Confident environment — risk 9% of account per trade
Neutral7%Mixed signals — be more conservative
Bear4%Risky environment — minimum sizing

The system calculates: contracts = floor((account_value × kelly_fraction) / max_loss_per_contract)

Risk Percentage Cap

Most experiments also enforce a simple "never risk more than X% on a single trade" rule. EXP-400 uses risk_pct = 8.5%. This acts as a maximum regardless of what Kelly says.

Maximum Positions Limit

Even if sizing math says "trade 10 contracts," the system won't open a new position if it already has too many open. EXP-600 (IBIT) caps at 5 open positions simultaneously.

Portfolio-Level Drawdown Circuit Breakers

The shared/portfolio_risk.py module tracks the combined equity across all Alpaca accounts and enforces portfolio-wide sizing limits:

LevelDrawdown from PeakAction
NORMAL > −8% Full sizing, business as usual
YELLOW −8% to −10% Reduce all new trade sizes by 50%
RED −10% to −12% Pause all new entries entirely
HARD STOP > −12% Flatten all positions immediately (in paper mode: log only)

7 Order Execution — Submitting to Alpaca

The execution layer is in execution/execution_engine.py (756 lines). It takes an approved opportunity and converts it into real brokerage orders.

The Execution Pipeline

1
Drawdown circuit breaker check
Before doing anything, the engine checks whether the account's drawdown has hit a circuit breaker level. If so, the order is rejected immediately with no API calls.
2
Duplicate check
Has the system already opened a similar position recently? Checks the database for existing "pending_open" records. Stale pending orders older than 60 minutes are automatically marked as failed so they don't block new trades indefinitely.
3
Market hours guard
Queries the Alpaca clock API to confirm the market is actually open. No orders on weekends or outside 9:30 AM – 4:00 PM ET.
4
Write to database FIRST (crash-safe design)
The trade record is written to SQLite in pending_open status before sending the order to Alpaca. This is crucial: if the system crashes after sending but before recording, the position would be "orphaned" — open with no record. By writing first, the reconciliation system can detect and recover it.
5
Build OCC option symbols
The system constructs the standardized option symbol format (called "OCC symbol"): 
# Format: TICKER + YYMMDD + C/P + 8-digit-strike-padded
# Example: SPY selling the $490 put expiring 2026-06-06
symbol = "SPY260606P00490000"
# Ticker + 260606 (Jun 6, 2026) + P (put) + 00490000 ($490 × 1000)
6
Submit the multi-leg order to Alpaca
Credit spreads are submitted as a single "MLEG" (multi-leg) order — both legs execute simultaneously at the net credit price. This prevents "leg risk" (one leg fills but the other doesn't). Iron condors are submitted as two MLEG orders (put wing + call wing separately). 
# Submit as one multi-leg order — both legs fill at once
order = alpaca.submit_mleg_order(
    legs=[
        {"symbol": "SPY260606P00490000", "side": "sell", "qty": 2},
        {"symbol": "SPY260606P00478000", "side": "buy",  "qty": 2},
    ],
    order_type="limit",
    limit_price=net_credit,   # mid-point of bid/ask
    time_in_force="day",
)
All orders are limit orders at the mid-price (halfway between bid and ask) to avoid overpaying the spread.
7
Deterministic order ID for idempotency
Each order gets a unique client_order_id derived from the trade's properties (ticker + strikes + expiration + millisecond timestamp). If the same order is accidentally submitted twice (e.g., due to a crash/restart), Alpaca will recognize the duplicate and reject the second one safely.
8
Error handling: never raises exceptions
The execution engine is designed to return a result dict instead of raising exceptions. Callers decide how to handle errors. If Alpaca rejects the order, the database record is updated to failed_open and the scan continues normally.

8 Position Management — Watching Trades After Entry

Getting into a trade is only half the battle. The PositionMonitor (execution/position_monitor.py, 1996 lines) is a background daemon that runs around the clock, watching every open position for exit signals.

The Three-Tier Reconciliation System

The monitor runs on three different cadences, each doing progressively more expensive checks:

TierFrequencyWhat it does
Tier 1 Every 60 seconds Check for pending fills, detect external closes (did something close outside the system?)
Tier 2 Every 5 minutes (market hours only) Full exit condition evaluation: stop loss, profit target, DTE management
Tier 3 EOD Once daily at 4:15 PM ET Process expiring positions, full reconciliation pass
Tier 3b Morning Once daily at 9:35 AM ET Catch overnight settlement activity (assignments, exercises)

Exit Conditions — When Does a Trade Close?

For each open position, the monitor evaluates these conditions in order:

A
DTE Management
If manage_dte is configured (e.g., close when ≤ 2 DTE), close the position before expiration to avoid gamma risk (options become very volatile in their final days).
B
Profit Target
Close when the position has gained a configured percentage of the maximum possible profit. EXP-400 closes at 55% of max profit. Example: collected $1.00 credit → close when the spread is worth $0.45 (you've kept 55% of the credit, locking in the win).
C
Stop Loss
Close when the spread has grown to a multiple of the original credit (the trade is losing). EXP-400 uses a 1.25× stop loss: if you collected $1.00 credit and the spread is now worth $2.25 (1.25× more than the $1.00 credit + the $1.00 credit itself = $2.25 cost to close), exit to limit the damage.
D
External close detection
If all legs of a position have disappeared from Alpaca (but the system still shows it as "open"), the monitor marks it as closed_external and estimates the P&L. This handles cases like early assignment, manual intervention, or API glitches.

Stale Close Order Protection

If the system tries to close a position but the close order is "stale" (submitted but not filled for too long), it automatically cancels and resubmits the order. It tries up to 3 times before flagging for manual review.

Early Assignment Detection

If unexpected stock positions appear in the Alpaca account (which can happen when short options are exercised early), the monitor detects these "equity ghost positions" and sends an alert.

9 The Experiments — 9 Parallel Strategy Tests

An experiment is one complete, isolated deployment of the trading system with a specific strategy configuration. Each experiment has:

  • Its own Alpaca paper trading account (isolated money)
  • Its own database file (data/pilotai_expNNN.db)
  • Its own config YAML (configs/paper_expNNN.yaml)
  • Its own subprocess running main.py scheduler

Think of it like running 9 different fund managers simultaneously, each with the same starting capital but different rules. The experiment protocol (defined in EXPERIMENT_PROTOCOL.md) requires a minimum 8-week paper trading window before any experiment is promoted to real money.

Why 9 Experiments?

Science requires controls. You can't know if a tweak (e.g., "what if we skip trades before NFP?") actually helps unless you run it in parallel with the original. The system is designed so new ideas are layered on top of the champion strategy, each tested independently, and only proven improvements get promoted.

10 Safety Systems — What Prevents Catastrophic Losses?

The system has multiple independent layers of protection. Each layer is designed to stop a different type of failure. They are intentionally redundant — if one fails, the next catches it.

Layer 1: Paper Mode Safety Lock

The function _validate_paper_mode_safety() in main.py runs every time the system starts. If paper_mode=true in the config but alpaca.paper=false, it raises an immediate fatal error and refuses to start. This prevents accidentally trading real money.

Layer 2: Regime Gate (EXP-3303b)

When the market is in "transition" or "high_stress" regime, SPY and QQQ entries are blocked entirely. This is a hard skip — not a size reduction. Implemented in shared/regime_gate.py.

Layer 3: VIX Hard Block

When VIX exceeds the configured maximum (e.g., VIX > 40 for EXP-800), the risk gate blocks all new entries. The market is in panic mode and options pricing is unreliable.

Layer 4: NFP Entry Filter (EXP-3311)

Skips all entries the day before Non-Farm Payrolls. The blacklist of NFP dates is published by the Bureau of Labor Statistics months in advance and stored in configs/event_blacklist.json.

Layer 5: Per-Trade Drawdown Circuit Breaker (Execution Engine)

The execution engine checks per-account drawdown before each order. If the account has fallen more than 40% from its peak equity, all new orders are blocked.

Layer 6: Portfolio-Level Circuit Breaker (4 tiers)

shared/portfolio_risk.py monitors combined equity across all accounts: −8% → halve sizing, −10% → pause entries, −12% → flatten all (in paper mode: log only). A 60-second cache prevents hammering the Alpaca API.

Layer 7: SENTINEL Guard (pre-scan check)

Before every scan, sentinel/guards.py runs a pre_scan_check() that verifies the experiment is in a valid state to trade. It checks experiment status, account health, and other preconditions. This is the system's "bouncer" — it can block a scan from even starting.

Layer 8: Write-Ahead Log (WAL) for Database Safety

SQLite runs in WAL (Write-Ahead Log) mode. This means database writes are extremely crash-safe — if the process crashes mid-write, the database is never left in a corrupted state. The shared/wal.py module provides additional recovery for DB write failures.

Layer 9: Watchdog Health Monitoring

railway_watchdog.py runs as a completely separate process on Railway. It:

  • Reads the .worker_status.json file every 300 seconds
  • Checks heartbeat files — if any experiment hasn't scanned in 45 minutes during market hours, it's suspicious
  • Tests Alpaca API connectivity
  • Checks database write recency
  • Sends Telegram alerts on any anomaly

Layer 10: Orphan Detection

The PositionMonitor runs daily orphan detection: it scans Alpaca for option positions that exist in the brokerage but have no corresponding record in the local database. These "orphaned" positions get synthetic records created so they can be managed properly.

The "No Synthetic Data" Absolute Rule

Rule Zero — Carlos Directive
All option pricing must come from IronVault (shared/iron_vault.py), which provides real historical option prices from a 276,000 contract database. Any code path that falls back to synthetic pricing — fixed credit fractions, Black-Scholes estimates used as prices, or BACKTEST_CREDIT_FRACTION — is classified as a critical bug.

IronVault hard-fails on startup if the options_cache.db database is missing or empty. There is no silent fallback. This ensures backtest results are based on prices that actually existed in the market, not mathematical approximations.

12 Infrastructure — Railway, Services, and Deployment

Railway Cloud Hosting

The system runs on Railway (railway.app), a Platform-as-a-Service that runs Docker containers from a Git repository. When code is pushed to the main branch, Railway automatically rebuilds and redeploys the application.

The Three Railway Services (from Procfile)

Service nameProcessPurpose
worker python railway_worker.py The master orchestrator. Reads the registry, spawns one scheduler subprocess per active experiment, and supervises them. Restarts crashed processes after 15 seconds.
watchdog python railway_watchdog.py Independent health monitor. Checks heartbeats, Alpaca connectivity, and database recency every 5 minutes. Sends Telegram alerts if anything looks wrong.
web uvicorn web_dashboard.app:app FastAPI web dashboard showing live P&L, open positions, and trade history. Accessible via a Railway-provided URL.

Persistent Volume

Railway provides a persistent disk volume mounted at a configurable path (set via the env var RAILWAY_VOLUME_MOUNT_PATH, e.g., /app/data). All SQLite databases live here. Without persistence, databases would be wiped on every deployment.

Each experiment writes to its own database file:

  • /app/data/pilotai_exp400.db — EXP-400 trades, positions, states
  • /app/data/exp503/pilotai_exp503.db — EXP-503 (in subdirectory)
  • /app/data/.worker_status.json — Shared status file between worker and watchdog
  • /app/data/heartbeat.json — Last scan heartbeat per experiment

Deployment Process

  1. Developer pushes code to main branch on GitHub
  2. Railway detects the push and starts a new build (Docker container)
  3. New container is deployed, Railway sends SIGTERM to the old one (graceful shutdown)
  4. The worker receives SIGTERM, terminates all experiment subprocesses cleanly, and exits
  5. New container starts: Procfile launches worker, watchdog, and web processes
  6. Worker reads registry.json, spawns experiment processes, and the system is live again
Zero-downtime during market hours
Deployments during market hours will briefly interrupt any in-flight scans, but the PositionMonitor state is recovered from the SQLite database on startup. Open positions are never lost — the startup reconciliation re-discovers them from Alpaca on the next boot.
