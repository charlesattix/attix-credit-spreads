Proposed Architecture

How the Four Runtime Gates Fit Into SENTINEL

The existing SENTINEL flow is: scan startup → pre_scan_check() → execute scan → exit. The runtime gates run after trade execution, either inline at scan end or on a separate scheduled cadence. They write to the same sentinel.db audit tables and emit Telegram alerts via shared.telegram_alerts.

┌─────────────────────────── Every Scan Cycle ────────────────────────────┐ │ │ │ [PRE-SCAN] [SCAN EXECUTION] [POST-SCAN (new)] │ │ │ │ Gate 1–5 main.py scan Gate 6: sizing check │ │ (static) exp800 scanner Gate 7: orphan detector │ │ ── orders placed ── Gate 9: lifecycle monitor │ │ │ │ [ROLLING WINDOW — every scan cycle accumulates data] │ │ Gate 8: Live-vs-Backtest Drift Tracker │ │ (checks rolling 30-trade window; fires if threshold breached) │ │ │ └──────────────────────────────────────────────────────────────────────────┘ Data Sources: trades DB ──→ Gate 6, 7, 8, 9 Alpaca API ──→ Gate 7 (position reconciliation) sentinel_state.json ──→ read by all gates; written by Gate 8 on halt sentinel.db ──→ alerts_log, experiment_snapshots written by all gates

Each gate calls from sentinel.guards import pre_scan_check as an existing pattern but does not replace it — the existing 5-gate foundation remains unchanged. Runtime gates are additive, non-breaking.

Runtime Gate
Gate 6
Trade Sizing Validator
Verify that actual contracts placed match the backtest sizing formula

What It Checks

Every trade placed in paper trading is sized by the formula: floor(account_equity × risk_per_trade / max_loss_per_spread). Live execution can diverge from this — order rounding, stale equity reads, concurrent position interference, or scanner bugs. Gate 6 computes the expected contract count for each newly placed trade and flags when actual sizing deviates by more than 15%.

Note: The trade_deviations table already exists in each experiment DB and tracks credit and P&L deviations. Gate 6 extends this pattern with a contracts-count column.

How It Works

  1. On scan completion, query trades where status = 'open' and entry_date ≥ scan_start_time (new trades only).
  2. For each new trade: fetch account_equity from Alpaca GET /v2/account (already called in Gate 3 health check — re-use the response).
  3. Reconstruct expected contracts: expected = floor(equity × risk_pct / (spread_width × 100)). risk_pct and spread_width read from the locked config fingerprint.
  4. Compare: deviation_pct = abs(actual − expected) / expected. If > 15%, emit alert.
  5. Write result to trade_deviations (new column: expected_contracts, actual_contracts, sizing_deviation_pct).

Data Sources

Inputs

  • trades table — contracts, entry_date
  • Alpaca /v2/account → equity
  • Paper YAML config → risk_per_trade, spread_width
  • sentinel_state.json → config_fingerprint (guard)

Outputs

  • trade_deviations — new sizing columns
  • alerts_log — on threshold breach
  • Telegram alert — warning or critical

Alert Thresholds

DeviationSeverityActionExample
0–15% OK Log only (normal rounding) Expected 9 contracts, placed 9
15–35% WARNING Telegram alert, log to alerts_log Expected 10 contracts, placed 7
> 35% CRITICAL Telegram alert + pause experiment on repeat Expected 10 contracts, placed 1
0 contracts placed, order accepted HALT Immediate halt — sizing formula broken Floor returned 0 due to equity read error

Implementation Complexity

Low–Medium ~180 lines of Python. Extends existing trade_deviations schema (1 migration). Alpaca equity call already happens in Gate 3 health check — re-use cached response to avoid double API call.

Integration Points

  • Add run_gate6_sizing_check(exp_id, new_trade_ids) call at end of every scanner's main loop.
  • Inject via same pattern as pre_scan_check(): two lines at the bottom of the scan execution block.
  • Schema migration: ALTER TABLE trade_deviations ADD COLUMN expected_contracts INTEGER; and two more columns.
Gate 7
Orphan / Unmanaged Position Detector
Real-time reconciliation of Alpaca positions vs DB state — every scan cycle

What It Checks

Gate 7 is the most urgent of the four proposals. It is the direct solution to the 99 unmanaged orphan trades currently invisible to SENTINEL.

A reconciler already exists at shared/reconciler.py (~1,400 lines) and runs three-tier reconciliation within each scanner. However, the reconciler operates within the scanner's own state — it has no channel to SENTINEL. Gate 7 is the SENTINEL-side consumer of reconciler output: it queries the DB immediately after each scan, identifies the delta, and fires alerts.

Two Failure Modes

🔴 Orphan

Position exists in Alpaca but not in the trades DB (or DB status ≠ open). Broker opened or retained a position the system doesn't know about. Risk: uncontrolled exposure accumulating silently.

Current examples: 14 unmanaged in EXP-400, 27 in EXP-503 — all have contracts=0 and short_strike=0, indicating leg-tracking failure at order submission.

👻 Ghost

Position exists in DB as status=open but not in Alpaca. Broker closed outside the system (manual close, margin call, corporate action, Alpaca-side expiration handling). Risk: system believes it has protection it no longer has; potential double-entry on next scan.

Current examples: 5 needs_investigation in EXP-401 match this pattern — DB open, Alpaca confirms closed.

How It Works

  1. Call Alpaca GET /v2/positions — returns all current open positions for the account.
  2. Query trades where status = 'open'.
  3. Compute set delta: orphans (Alpaca ∖ DB), ghosts (DB ∖ Alpaca).
  4. For orphans: create a placeholder trades record with status='unmanaged', fire WARNING alert. Do NOT close — hands-off. Gate 7 detects, humans resolve.
  5. For ghosts: update trade to status='needs_investigation', fire CRITICAL alert.
  6. Track consecutive-scan count: if same orphan persists for ≥ 3 scans without resolution, escalate severity.

The reconciler already does this at Tier 2 (~5 min) and Tier 3 (EOD). Gate 7 does not replace it — it gives SENTINEL independent visibility and ensures the alerts_log is always up to date.

Data Sources

Inputs

  • Alpaca /v2/positions — live broker positions
  • trades table — status='open' records
  • scanner_state — last_orphan_check timestamp
  • reconciliation_events — prior reconciler actions

Outputs

  • trades — new unmanaged stub records for orphans
  • alerts_log — one row per detected orphan/ghost
  • scanner_state.last_orphan_check — updated
  • Telegram: immediate alert with count + position IDs

Alert Thresholds

ConditionSeverityTelegram Message
New orphan detected (1st scan) WARNING ⚠️ EXP-400: 1 new orphan position detected. Symbol: SPY. Investigate within 24h.
Same orphan persists ≥ 3 scans CRITICAL 🛑 EXP-400: orphan position unresolved for 3+ scans. Manual close required.
Ghost position detected CRITICAL 🛑 EXP-400: DB shows open trade but Alpaca has no position. Possible double-entry risk.
≥ 5 orphans simultaneously HALT 🛡️ SENTINEL — EXP-400 HALTED. 5+ untracked positions. Scanner blocked until resolved.

Implementation Complexity

Low ~150 lines. Logic is nearly identical to the reconciler's Tier 2 check — Gate 7 is a read-only SENTINEL wrapper around work that already runs. Primary effort: consuming reconciliation_events to avoid duplicate alerts.

Why Not Just Use the Reconciler?

The reconciler operates within each scanner process and logs to its own reconciliation_events table. SENTINEL has no subscription to that table and no cross-experiment view. Gate 7 provides: (1) cross-experiment orphan count visibility, (2) escalation when reconciler fails to clear an orphan, and (3) automatic HALT if orphan count exceeds safe threshold.

Gate 8
Live-vs-Backtest Drift Tracker
Rolling statistical comparison of live performance vs backtest expectations

What It Checks

Gate 8 answers the question SENTINEL currently cannot: "Is this experiment still trading like the backtested strategy?" It tracks three rolling metrics over the last 30 closed trades, comparing each to the certified backtest baseline.

This gate is what would have caught EXP-400's win rate collapsing to 57% (21pp below the 78% backtest expectation) and triggered a review before additional capital was deployed into stop-loss conditions.

How It Works

  1. After each scan, query last 30 closed trades from the trades DB.
  2. Compute: win rate, avg loss per losing trade, peak-to-current drawdown from scanner_state.peak_equity.
  3. Compare against thresholds stored in sentinel_state.json under a new backtest_baseline key (written at certification time by certify_all.py).
  4. Write rolling snapshot to experiment_snapshots table in sentinel.db.
  5. On threshold breach: log to alerts_log, Telegram, and — on HALT level — call state.set_halt(exp_id).

Minimum Trade Sample Size

Win rate is not meaningful over fewer than 10 trades. Gate 8 uses a minimum 10-trade sample before firing win rate alerts, and downgrades all alerts by one severity level for fewer than 20 trades. This prevents false alerts on new experiments.

Implementation Complexity

Medium ~300 lines. Requires: (1) adding backtest_baseline block to sentinel_state.json schema — populated at certification time, (2) writing rolling window query, (3) hooking into post-scan execution flow. The experiment_snapshots table in sentinel.db already exists and accepts win_rate/total_pnl fields.

State Persistence

Rolling window state is re-computed from the DB on each run — no in-memory state required. This means the gate is crash-safe and correct after scanner restarts. Alert deduplication uses the existing alert_dedup table in each experiment DB.

Gate 9
Position Lifecycle Monitor
Time-based enforcement: every open position must progress or be explained

What It Checks

Gate 9 enforces time-bounded position lifecycle. Credit spread strategies have a defined lifecycle: open → managed (PT or SL hit) → closed. Any trade stuck in an intermediate state for more than a threshold duration is a silent failure — the strategy is not being managed as designed.

Currently there are 2 pending_close trades in EXP-400 from 2026-04-02 that have been stuck for 15 days with no resolution. SENTINEL never noticed.

Monitored States and Thresholds

State Description Warning After Critical After Severity
pending_open Order submitted but not confirmed filled 30 min 2 hours WARNING → CRITICAL
pending_close Close order submitted but not confirmed 30 min 2 hours CRITICAL
needs_investigation Reconciler flagged — requires manual review 4 hours 24 hours CRITICAL → HALT
open past DTE expiry Option expired but trade still shows open Immediate Next scan HALT
unmanaged (new) Orphan detected in Gate 7 1 scan 3 scans without resolution WARNING → CRITICAL
Current Violation: EXP-400 pending_close (2026-04-02)
Two trades stuck in pending_close since 2026-04-02 — 15 days past threshold. Under Gate 9, this would have triggered CRITICAL within 2 hours of the close order being placed and escalated to HALT on the following scan.

How It Works

  1. On every scan: query all trades where status IN ('pending_open', 'pending_close', 'needs_investigation').
  2. For each: compute age_hours = (now − entry_date or updated_at).
  3. Apply threshold ladder. Update alerts_log if threshold crossed.
  4. Separately: query all status='open' trades where expiration < today. These are expired-but-open, requiring immediate action.
  5. Alert deduplication: suppress re-alert for same trade_id at same severity within 4 hours using alert_dedup table.

Data Sources

Inputs

  • trades — status, entry_date, expiration, updated_at
  • alert_dedup — deduplication
  • Wall clock (scan timestamp)

Outputs

  • alerts_log in sentinel.db
  • Telegram alerts
  • sentinel_state.json status → halted on severe cases

Implementation Complexity

Low ~120 lines. Purely time-based DB queries — no external API calls required. Can share the alert dedup infrastructure already present in each experiment DB.
