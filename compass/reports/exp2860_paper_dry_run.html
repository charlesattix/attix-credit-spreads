Verdict

PIPELINE GREEN — every built order passes schema validation; mock submission returned 'accepted' for every leg. Ready for real Alpaca keys.

Methodology & caveats

  • Signal source. compass.exp2830_paper_signal_generator.generate_all_signals called verbatim. Uses real Yahoo market data.
  • VIX ladder. compass.exp2820_flash_crash_protection.VIX_LADDER applied as a size multiplier on the EXP-2830 per-stream target_contracts. Signals whose post-ladder size rounds to 0 are marked VIX_FLAT.
  • Alpaca schema. The order payloads follow Alpaca's documented multi-leg options spec: top-level order_class='mleg', legs array with ratio_qty and position_intent per leg, OCC option symbols. Validation checks the required keys and per-leg schema but does NOT validate that the strikes actually exist on the exchange — that is Alpaca's job at real submission.
  • Mock submission. No network call is made. The mock response object returns the shape we expect from a real /v2/orders POST — id, client_order_id, status='accepted', legs. It is marked with _mock: true so downstream consumers know it is not a live fill.
  • cross_vol sleeve. EXP-2020's IV−RV dispersion trade is a model sleeve, not a single underlier + strike. The dry run emits a sentinel _CROSS_VOL_MODEL_ leg with a NEEDS_MODEL_EXEC note so the pipeline stays consistent; production will route this sleeve through its own execution model rather than Alpaca multi-leg.
  • What this does NOT prove. (a) Real Alpaca account mechanics (option permissions, buying-power checks, margin); (b) fill quality (no market data hit the order book); (c) risk-management integration (position limits, circuit breakers, kill switch wiring). Those are separate smoke tests that need the real API keys.
